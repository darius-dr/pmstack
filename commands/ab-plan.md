# /pm:ab-plan

You are dispatching the A/B Test Planner subagent.

## Procedure

1. **Parse arguments**:
   - First positional: section to test (`hero`, `pricing`, `cta`, etc.).
   - `--traffic`: weekly sessions (if not provided, ask user — the planner should refuse to scaffold low-power tests).
   - `--baseline-conv`: current conversion rate as a decimal (e.g. 0.018). If unknown, ask.

2. **Refuse below traffic threshold**. If `weekly-sessions < 1000` (a reasonable floor for hero tests at typical baseline rates), tell the user explicitly: "Insufficient traffic for an A/B test on this section. Ship the best variant and instrument click-through; revisit when you have ≥1000 weekly sessions." Stop.

3. **Dispatch the A/B Test Planner subagent** (Task tool, `subagent_type: ab-test-planner`) with this brief:

   ```
   Plan an A/B test for the {section} section.

   Context:
   - Current copy: {paste from repo}
   - Weekly sessions on this section: {N}
   - Baseline conversion: {rate}
   - Audience segments documented: {list}
   - Variants generated by /pm:copy (if any): .pmstack/copy/{section}-{segment}.md

   Deliverables:

   ## Hypothesis
   One sentence per variant. Format: "If we change {X}, then {primary metric} will {direction} because {causal theory rooted in ICP}."

   ## Variants
   For each (A = control + 2 challengers max):
   - Headline / sub / CTA differences
   - The single design or copy lever being tested
   - Expected lift range (conservative-optimistic, with reasoning)

   ## Primary metric
   - One conversion event (NOT a click — a real conversion downstream).
   - Secondary metrics (≤2): leading indicators.

   ## Power calculation
   - Minimum detectable effect (MDE) at α=0.05, β=0.2
   - Sample size per variant
   - Estimated test duration at current traffic
   - If duration > 4 weeks: flag as "too long to power" and propose a different variant with bigger expected lift.

   ## Guardrails
   - Metrics that must NOT regress (bounce rate, scroll depth, etc.).
   - Stopping rules.

   ## Tracking implementation
   - Concrete event names + properties the engineering team should fire.
   - Where in the funnel each event lives.

   Output to .pmstack/ab/{section}-{YYYY-MM-DD}.md.

   Hard rules:
   - Never test more than 2 challengers vs control.
   - Never test more than one lever per variant (otherwise you can't attribute the lift).
   - Refuse tests below the documented traffic threshold.
   - If the variant would require fabricated proof to land, mark it [BLOCKED: needs proof] and stop.
   ```

4. **Report** to the user: variant headlines, expected duration, and one risk to watch.
